## 实现时间

2026-02-13

## 实现的功能

本次实现了两个相互关联的功能：

### 1. 文档健康检查（已完成）

**功能描述:** 在 brainstorming 开始时检查项目文档是否过时

**实现方式:**
- 修改 `skills/brainstorming/SKILL.md`
- 添加 "Pre-Check: Documentation Health" 章节
- 询问用户是否需要更新文档
- 可选调用 code-structure-reader 技能

**相关文件:**
- ✅ `skills/brainstorming/SKILL.md` (已修改)
- ✅ `docs/feature-documentation-check-integration.md` (功能说明)

### 2. 模块复用设计（已完成）

**功能描述:** 在技术方案设计时自动检查现有模块，避免重复开发

**实现方式:**
- 创建新技能 `design-with-existing-modules`
- 集成到 brainstorming 的技术设计环节
- 提供三种决策模式：复用/改造/新增
- 强制同步更新项目文档

**相关文件:**
- ✅ `skills/design-with-existing-modules/SKILL.md` (新建，441行)
- ✅ `skills/brainstorming/SKILL.md` (已修改)
- ✅ `docs/feature-module-reuse-design.md` (功能说明)
- ✅ `docs/module-reuse-verification.md` (验证清单)

## 关键特性

### 1. 非阻塞式设计

两个新功能都不会强制用户：
- 文档更新检查可以跳过
- 模块复用分析是最佳实践，不是强制要求

### 2. 自动化决策支持

提供清晰的决策树：
```
相似度 ≥ 95%  → REUSE
相似度 60-94% → MODIFY
相似度 < 60%  → CREATE NEW
```

### 3. 文档同步强制

对于 MODIFY 和 CREATE NEW 的决策，强制要求：
- 更新对应的项目文档
- 遵循已有格式
- 标注变更内容

### 4. 完整的格式指导

提供详细的格式示例：
- 设计文档中的决策记录格式
- 项目文档中的更新格式
- 改造方案描述模板

## 使用示例

### 示例 1: 完整的复用场景

```bash
# 1. 生成项目分析文档
/brainstorm
→ 选择 "Documentation needs update"
→ code-structure-reader 生成 11 个文档

# 2. 设计新功能
"帮我设计用户登出功能"
→ brainstorming: Part 1 完成
→ design-with-existing-modules 分析
→ 发现 AuthService 已有 logout() 方法
→ 决策: ✅ REUSE 100%
→ 设计文档标注直接复用

# 3. 无需更新项目文档（因为是 REUSE）
```

### 示例 2: 改造场景

```bash
# 1. 设计新功能
"帮我设计用户头像上传功能"
→ design-with-existing-modules 分析
→ 发现 UserService 存在，70% 匹配
→ 决策: 🔧 MODIFY
→ 设计文档包含:
  - 从: 仅支持文本资料
  - 到: 支持头像上传
  - 变更列表: uploadAvatar(), deleteAvatar()
  - 影响分析: 无破坏性变更

# 2. 更新项目文档
→ 更新 docs/project-analysis/03-backend-domains.md
→ 在 UserService 中添加新方法
→ 在 User 实体中添加 avatar_url 字段
```

### 示例 3: 新增场景

```bash
# 1. 设计新功能
"帮我设计订单退款功能"
→ design-with-existing-modules 分析
→ 未找到任何退款相关模块
→ 决策: ➕ CREATE NEW
→ 设计文档包含:
  - 新增理由: 无现有功能
  - 功能描述: 退款申请、审核、执行
  - 集成点: OrderService, PaymentService

# 2. 更新项目文档
→ 在 03-backend-domains.md 中添加 RefundService 章节
→ 在 04-database-schemas.md 中添加 refund 表
→ 在 02-backend-apis.md 中添加退款接口
```

## 技术优势

### 对开发者

1. **节省时间**
   - 复用: 节省 80-100% 时间
   - 改造: 节省 30-50% 时间
   - 新增: 清楚知道为什么必须新建

2. **提高质量**
   - 基于现有架构设计
   - 保持代码风格一致
   - 复用经过测试的代码

3. **降低风险**
   - 避免破坏现有功能
   - 明确改造影响范围
   - 及早发现潜在问题

### 对项目

1. **减少技术债务**
   - 避免重复实现
   - 保持架构一致
   - 提高可维护性

2. **文档与代码同步**
   - 强制更新文档
   - 确保文档准确
   - 降低 onboarding 成本

3. **知识沉淀**
   - 记录设计决策
   - 说明改造理由
   - 便于后续审查
