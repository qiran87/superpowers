# Bug 修复工作流 (Debugging Workflow)

本文档说明 Superpowers 中三个核心调试技能的协作关系和完整工作流程。

## 核心技能关系

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         systematic-debugging                             │
│                            (主控技能)                                    │
│                     控制整个调试流程和循环计数                            │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │                    test-driven-development                        │ │
│  │                         (测试驱动)                                  │ │
│  │                    RED → GREEN → REFACTOR                          │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│           ↑                                                              │
│           │ 被调用                                                      │
│           │                                                              │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │                verification-before-completion                     │ │
│  │                          (验证)                                    │ │
│  │              运行验证命令 → 确认输出 → 声明结果                   │ │

│           ↑                                                              │
```

## 循环限制规则

### 最大尝试次数：7 次

**计数规则：**
- 每次 Phase 4 完整执行 Steps 1-4 = 1 次尝试
- 如果 `verification-before-completion` 验证失败，计数 +1
- 如果验证通过，循环结束，调试成功

**达到 7 次后的强制措施：**

```
┌─────────────────────────────────────────────────────────────┐
│ ⚠️  已达到 7 次调试尝试上限                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ 这表明可能是架构问题，而非单个 bug：                        │
│                                                             │
│ • 每次修复都在不同地方引发新问题                            │
│ • 修复需要大规模重构                                        │
│ • 修复创造了其他地方的副作用                                │
│                                                             │
│ 🛑 停止修复，质疑基础架构：                                  │
│                                                             │
│ 1. 这个模式根本上是否合理？                                 │
│ 2. 是否因为惯性而坚持？                                     │
│ 3. 应该重构架构还是继续修补症状？                           │
│                                                             │
│ 💬 与人类伙伴讨论后再继续                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 为什么是 7 次？

| 次数 | 含义 |
|------|------|
| 1-2 | 正常调试，初次尝试 |
| 3-4 | 深入调查，可能涉及多个因素 |
| 5-6 | 复杂问题，接近架构边界 |
| **7** | **临界点：很可能是架构问题** |

## 技能调用顺序

### Phase 4 中的标准顺序

```python
# 伪代码表示调试循环
attempt = 0
max_attempts = 7

while attempt < max_attempts:
    attempt += 1
    print(f"=== 调试尝试 {attempt}/{max_attempts} ===")

    # Phase 1-2: 定位问题
    root_cause = investigate_root_cause()
    pattern = analyze_pattern()

    # Phase 3: 假设验证
    hypothesis = form_hypothesis()
    test_result = test_minimally(hypothesis)

    # Phase 4: 实施修复
    # Step 1: 创建失败测试
    use_skill("test-driven-development")  # RED

    # Step 2: 实施修复
    use_skill("test-driven-development")  # GREEN

    # Step 3: 初步验证
    if not test_passes():
        continue

    # Step 4: 验证修复
    verification = use_skill("verification-before-completion")

    if verification.success:
        print("✅ 调试成功!")
        break
    else:
        print(f"❌ 验证失败，尝试 {attempt}/{max_attempts}")

if attempt == max_attempts:
    print("⚠️  达到最大尝试次数，请质疑架构设计")
```

## 验证方式说明

### verification-before-completion 支持的验证方式

| 验证类型 | 命令格式 | 适用场景 |
|---------|---------|---------|
| **单元测试** | `npm test`, `pytest` | 功能验证 |
| **HTTP API** | `curl -X POST http://...` | API 端点测试 |
| **Python 脚本** | `python verify_fix.py` | 复杂验证逻辑 |
| **Shell 命令** | `grep`, `pgrep`, `netstat` | 系统状态验证 |
| **集成测试** | 多步骤 bash 脚本 | 端到端验证 |
